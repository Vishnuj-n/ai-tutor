# Walkthrough: Startup Race Condition Synchronization

We have fixed the startup race condition where the frontend initiates API requests (`GetUserSettings` and `IsOnboarded`) before the Go backend's asynchronous bootstrapping procedure (`runtime.Bootstrap(ctx)`) finishes initializing the database repository (`a.repo`), preventing nil pointer dereference panics.

## Key Changes

### 1. App Synchronization Channel
- **`readyChan` Integration**: Added a coordination channel `readyChan chan struct{}` to the thin Wails [App](file:///c:/Users/vishn/PROJECT/ai-tutor/app.go#L35-L52) bridge.
- **Initialization**: Initialized the channel in [NewApp](file:///c:/Users/vishn/PROJECT/ai-tutor/app.go#L54-L58) and deferred its closing inside the [startup](file:///c:/Users/vishn/PROJECT/ai-tutor/app.go#L66-L91) method. This ensures that callers waiting on the channel are unblocked immediately once startup finishes (either successfully or in error state).
- **Helper Method**: Introduced `waitForReady()` which blocks callers by checking/waiting on `readyChan` if initialized, allowing test suites initializing `&App{}` directly without channels to run immediately.

### 2. Guarding Frontend Entrypoints
- **Blocked Calls**: Added `waitForReady()` at the top of [GetUserSettings](file:///c:/Users/vishn/PROJECT/ai-tutor/app.go#L787-L808) and [IsOnboarded](file:///c:/Users/vishn/PROJECT/ai-tutor/app.go#L1099-L1110).
- **Safe Nil Checks**: If the database bootstrapping completely fails, `a.repo` remains `nil`. The handlers now check for a nil repository and return structured error maps rather than dereferencing the pointer and crashing.
